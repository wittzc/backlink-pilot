# Launch 平台半自动注册 — 跨会话接手笔记

> 本会话(2026-06-04)验证了"登录态复用"可行,但没写 recipe。下个会话接手:注册
> launch 平台 + 保留登录态 + 验证产品提交表单 → 标准则写 recipe。

## 背景

用户给的 47 个首发站里,18 个是要**登录账号**的 launch 平台(targets.yaml 分组
`overseas_launch_2026_06`,`auto: manual`)。思路:用一个**专门的 Google 账号**手动注册各站,
bb-browser 复用登录态后**半自动填产品提交表单**(不追求全自动批量)。

## 本会话已验证

1. **登录态复用成立** ✅ —— 用户在 bb-browser 启动的 Chrome 里手动 Google 登录 TinyLaunch 后,
   `bb-browser snapshot` 看到导航栏 `Sign In` 变 `Dashboard` + `Sign Out`,证明 bb-browser
   操作的 Chrome 带着登录态。**「登一次 → 工具复用」整条地基通了。**
2. **bb-browser 启动方式**(踩过坑):daemon 会卡 `HTTP 503 Chrome not connected`,
   修复 = `bb-browser daemon shutdown` 后再 `bb-browser open about:blank`。环境 bb-browser 0.11.3。

## TinyLaunch 具体发现(关键,省下个会话重新踩)

- `submit_url` 填的 `https://tinylaunch.com/submit` 是 **404** —— agent 评估给的 URL 不准,实测才暴露。
- 首页 "Submit Product" 是 **React button**(`ref=9`),`bb-browser click` 点了**没反应**
  (React onClick 不响应命令行 click,要 `evalClickReal`,见 troubleshooting.md「React forms」)。
- **登录后 dashboard 内表单是标准 HTML** —— `tinylaunch.com/dashboard/profile`(Maker Profile:
  First Name / Last Name / Handle / Twitter)是普通 input,bb-browser 能 `fill`。
- 真正提交产品在左侧 **Startups** 菜单(带🔒,要先填 Maker Profile 解锁)—— **本会话未验证
  Startups 的产品提交表单结构**,这是下个会话第一件事。

## 判断基线(别忘)

- 全自动批量发 launch 平台 ROI 低(每站结构不同 + React + 多步 + 封号风险),**不挑站全自动 = 亏**。
- 半自动(登录态复用 + 标准表单 fill)对**标准表单的站**才值。挑简单的做,React 多步的手动发。
- 这条对应 METHODOLOGY 原则 5(fail-fast 不硬突破)+ ROI 优先,不是技术做不到,是不值得硬啃。

## 用专门 Google 账号

注册这些站**用一个专门 Gmail**,别用主账号 —— launch 平台对批量行为敏感,被风控时牺牲小号
(troubleshooting.md「Email for Agent Registration」建议过 agent 专用 Gmail)。
