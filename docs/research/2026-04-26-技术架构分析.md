## 一、项目定位

**一句话**：面向独立开发者的「外链提交自动化工具包」——一个 config，一条命令，把产品批量提交到 250+ 目录站。

**关键事实**：
- Node 18+ ESM 项目，纯 CLI，无服务端、无数据库
- v2.2.0，作者 sagasu，MIT 协议，仓库 [s87343472/backlink-pilot](https://github.com/s87343472/backlink-pilot)
- 引擎策略：**bb-browser 唯一引擎**（真实 Chrome 子进程，反检测能力强）；playwright 已在 v2.2 物理移除
- 设计哲学：**CLAUDE.md 即说明书**——目标用户用 `claude` 直接对话即可，不需要读文档

## 四、关键数据流

### 4.1 单站提交 `submit saashub --engine bb`

```
cli.js
  → loadConfig()                        [校验 product.* 必填]
  → submit.js:submit('saashub')
      → loadAdapter('saashub')          [动态 import sites/saashub.js]
      → 预检 HTTP（404 → PAGE_404，5xx → SERVER_ERROR）
      → startMs = Date.now()
      → withBrowser({engine:'bb'})      [拉起 bb-browser]
      → adapter.submit(product, config)
          → 登录 / 填表 / humanType / 截图 / 点击
      → duration_ms = Date.now() - startMs
      → recordSubmission()              [proper-lockfile → atomic write → submissions.yaml]
      → 失败时: classifyError() → { code, nextSteps[] } → 渲染 / JSON 输出
```

### 4.2 批量博客评论 `node src/batch-submit.js --limit N`

```
读 resources/backlink-resources.json
  → 优先级打分（blog_comment > profile，has_url > 无，no_captcha > captcha）
  → 过滤 logs/global-history.json 已提交项
    （v2.2: loadAllHistory() 也检查 submissions.yaml，双轨去重）
  → 逐条 submitBlogComment（5 人格轮换，模板生成评论）
  → 写 logs/submissions-{DATE}.json + global-history.json
  → 每条间隔 15-45s 随机
```

### 4.3 站点适配器接口契约

所有 [src/sites/*.js](../../src/sites/) `default export` 同一形状对象：

```js
{
  name,
  url,
  auth,        // none | email-password | oauth
  captcha,     // none | color | turnstile
  engine?,     // bb（v2.2 唯一合法值）
  submit(product, config) → { url, confirmation }
}
```

`generic.js` 是兜底——通过 bb-browser 的 a11y snapshot 自动识别 name/url/email/description 字段。截图命名 `{hostname}-{date}.png`，同站同日覆写。

**结构示例**：

```yaml
- name: Future Tools
  submit_url: https://www.futuretools.io/submit-a-tool
  type: form
  auto: yes
  lang: en
  notes: Popular AI tool directory
```

**设计决策**：targets.yaml 不被代码直接消费——它是给 Claude / 人读的清单，由 Claude 按规则筛选后逐个调用 `submit`。这是 agent-first 设计的体现。加 web UI 需重构为数据库消费。

## 七、设计亮点

1. **bb.js 的 `evalClickReal`**：不靠 Playwright `.click()`，而是在浏览器内派发完整 `mousedown → mouseup → click` 序列——绕过 React/Vue 组件忽略合成事件的问题。反爬攻防实战经验沉淀。

2. **CLAUDE.md 即 UI**：项目把 agent 当成一等公民——README 里最快入口是 `claude` 而非 CLI。targets.yaml 设计成 YAML 而非 DB，就是为了让 Agent 能直接读筛。v2.2 加了 Batch Submission Playbook，进一步强化 agent 路径。

3. **预检机制**：提交前先 HEAD 一次 URL，避免拉起浏览器才发现 404，省时间 + 不留指纹。v2.2 将预检结果映射为结构化 error code。

4. **nextSteps 数据契约**（v2.2 新增）：每次失败返回 `{ code, nextSteps[] }`，CLI 渲染可读版，`--json` 模式 Claude agent 可解析并自动执行下一步。

5. **批量人格轮换**：5 个虚拟身份 + 20+ 评论模板 + 15-45s 抖动——博客评论场景的反检测套路。

6. **parseDocument 保留注释**（prune-dead / mark-*）：用 yaml 库的 `parseDocument()` 而非 `parse()` + `stringify()`，targets.yaml 头部 14 行注释在修改后完整保留。

## 九、用户视角

### 9.1 典型用户画像

| 画像 | 描述 | 目标 | 痛点 |
|------|------|------|------|
| **A — Indie Hacker（主力）** | 1 人开发者，刚上线 SaaS / 工具站，没 SEO 预算 | 让产品在 Google 上能被搜到，最低成本拿外链 | 不想花几小时手动一个个站填表；不懂哪些站值得提交；怕被封号 |
| **B — 中小团队 marketer** | 2-5 人小团队的运营，负责 SEO 冷启动 | 系统化跑通 100+ 站点提交，留下凭证可复盘 | 重复劳动；提交记录散落 Excel；漏交 / 重交 |
| **C — Agent 玩家** | 熟 Claude Code / OpenClaw，喜欢「说一句话就跑」 | 把外链当成 agent 任务编排出去 | 工具不是 agent-friendly（要读长文档、参数复杂） |
| **D — 二开开发者** | 想给自己的站做适配器、贡献给项目 | 快速理解适配器接口，复用 bb-browser 能力 | 缺 adapter 模板示例；不懂 bb-browser 反检测细节 |

### 9.2 用户旅程

#### 旅程一：Claude Code 用户（推荐路径，最短）

```
[发现] GitHub 看到 README → 注意到「don't need to read any docs」
   ↓
[安装] git clone + npm install + bb-browser → 30 秒
   ↓
[启动] 终端里 cd 项目 → 输入 `claude`
   ↓
[对话] 「帮我提交外链」
   ↓ Claude 读 CLAUDE.md → 触发 First-time User Flow
[配置] Claude 问 5 个问题 → 自动生成 config.yaml
   ↓
[启动浏览器] Claude 跑 `bb-browser open about:blank`
   ↓
[首次试水] Claude 跑 `submit futuretools --engine bb` 验证链路
   ↓
[批量] 用户说「提交到所有免费站」→ Claude 按 Batch Playbook 循环提交
   ↓
[复盘] `node src/cli.js status` + `stats --timing` + screenshots/
```

**体验关键节点**：
- ✅ 「3 步上手」承诺兑现：clone / install / claude
- ✅ Claude 主动问问题，用户不需要查文档字段
- ✅ v2.2 失败时 Claude 可解析 nextSteps 自动决策（mark-dead / 重启 Chrome）
- ⚠️ bb-browser 必须先 `open about:blank` 启动 Chrome——对非技术用户仍是黑盒
- ⚠️ Cloudflare Turnstile 需要 2 分钟内完成，慢站会失败

#### 旅程二：纯 CLI 用户（无 agent）

```
[配置] cp config.example.yaml config.yaml + 手动编辑
   ↓
[自检] node src/cli.js doctor → 确认环境 OK
   ↓
[运行] node src/cli.js submit futuretools --engine bb
   ↓
[失败处理] 看 nextSteps → 复制命令执行（v2.2 改善）
   ↓
[批量] 手动循环提交，或告知 Claude 按 Batch Playbook 跑
```

#### 旅程三：二开开发者

```
[侦察] node src/cli.js scout https://newsite.com --deep → 看字段
   ↓
[尝试通用] node src/cli.js submit https://newsite.com --engine bb
   ↓ 通用适配器失败（字段名特殊 / 多步表单）
[写适配器] 参考 src/sites/saashub.js → 写 newsite.js
   ↓
[测试] 跑一次提交 → 看 screenshots/ 验证
   ↓
[贡献] PR 到上游
```

**摩擦点**：docs/contributing.md 有规范但缺**最小可运行模板**；scaffold-adapter 命令未实现。
