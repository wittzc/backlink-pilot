## 当前结论

这批站点里，最高命中的入口不是“看起来像投稿页”的页面本身，而是它背后的真实表单或提交接口。

目前最有效的三类路径：

1. 第三方真表单：`Tally`、`Jinshuju`、`MikeCRM`、`Wufoo`
2. 纯 HTML 表单，可直接 `POST`
3. 前端页面调用公开 JSON API，可直接请求

目前最该跳过的五类路径：

1. `reCAPTCHA` / `captcha`
2. 登录墙
3. 付费墙
4. `badge` / `backlink` 强制要求
5. 强制上传 logo / 截图

## 浏览器使用经验

### Safari

问题：

- 标签页容易跳到 `auth.openai.com`
- 上传文件时不稳定
- 可访问性树偶尔把值映射到错误字段
- 标签切换后偶尔丢焦点

结论：

- 适合简单 `Tally` / `Jinshuju`
- 不适合上传重、交互复杂的表单

### Chrome

结论：

- 更适合处理真实第三方表单
- 多标签页检查结构更稳
- 但如果页面本身是 Bubble / 登录态噪音很重，也不值得硬打

### 浏览器选择的实际规则

只有在以下情况才值得启用浏览器自动化：

1. 直投 API 不明显
2. 表单是真实的第三方提交页
3. 没有验证码
4. 没有强制上传

## 上传文件阻断

新目录站里，强制上传 logo / 截图是常见阻断。

典型案例：

- `http://toolsdar.mikecrm.com/nix44in`
  - 表单真实可填
  - URL 甚至会自动预填
  - 但 `网站LOGO` 是必填

实际规则：

- 看到强制上传，优先级立刻下降
- 先打非上传型站点
- 只有当上传型目录站明显高价值时，再回头处理

## 识别规则

### 好信号

- 普通 `<form method="post">`
- JS 里有明确的 `fetch(...)`
- 字段名清楚
- 第三方表单链接真实可打开
- 防刷只是简单算术题

### 坏信号

- `recaptcha`
- `turnstile`
- `sign in`
- `badge`
- `upload`
- `price`
- `page not found`
- `typeform is now closed`

## 后续优先级

后续最值得继续打的目标：

1. 字段可见的纯 HTML 表单
2. 前端 JS 能直接抠出 API 的页面
3. `url + email` 这类老式目录提交页
4. 无验证码的 `Tally` / `Jinshuju` / `MikeCRM` / `Wufoo`

后续最不值得继续耗时的目标：

1. 付费提交流程
2. `reCAPTCHA` 页面
3. 已失效的 `Typeform` / `Airtable`
4. 强制挂 badge / backlink 的目录站
5. Bubble 类型但提交工作流不透明的页面
