# 站点画像驱动的外链内容差异化 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use executing-plans to implement this plan task-by-task.

**Goal:** 让 backlink-pilot 给同一产品向不同目录站提交时，按站点画像（`niche`）产出差异化的卖点角度 / 话术 / 语言，提升外链通过率；文案由运行时 agent（Claude Code / Codex）生成，零额外 API 成本，代码只负责供料、填充通道与 fallback。

**Architecture:**
- **数据层**：`targets.yaml` 每站加一个 `niche` 字段（6 枚举），复用已有 `lang`。AI 导航 / awesome / 社区分组走规则映射，~112 个 general 站用初级模型（haiku）批量细分。
- **接口层**：`submit` 支持外部传入文案（`--description-file <path>`）。有则覆盖提交文案，无则回退 `config.product`（纯 CLI 用户不退步）。外部文案经 `product.submit_text` 在 generic 与 recipe 两条填充路径统一生效。
- **agent 层**：`AGENT_GUIDE.md` 增「按 niche 生成文案」playbook（0 代码）。批量提交时每站读画像 → 生成 → 写临时文件 → `submit --description-file`。
- **反馈闭环**：本期不做，列为 Non-goal + TODO。

**Tech Stack:** Node ESM、`yaml`（parseDocument 保留注释格式）、commander、bb-browser、`node --test`。

**验证方式（必填，必须可观测）:**
- `grep -c "    niche:" targets.yaml` 返回 `258`（每站都有 niche）。
- niche 值只在白名单内：`grep -E "^\s+niche:" targets.yaml | grep -vE "ai-tools|saas|devtools|startup|community|general|design"` 无输出。
- 规则映射正确：`overseas_ai_directories` 下任一站点 niche 为 `ai-tools`；`awesome_lists` 下为 `devtools`；`reddit` 下为 `community`。
- 外部文案生效：`printf 'EXTERNAL TEXT' > /tmp/d.txt && node src/cli.js submit https://example.com/submit --description-file /tmp/d.txt --dry-run` 输出的 product JSON 中 `submit_text` 等于 `EXTERNAL TEXT`。
- fallback 生效：不带 `--description-file` 跑同命令 `--dry-run`，输出 product JSON **不含** `submit_text` 键。
- `npm test` 输出 `# pass`（数）= 原 226 + 新增用例数，`# fail 0`。
- `AGENT_GUIDE.md` 含「Niche-driven content」段，且 6 个 niche 每个都有「强调什么 + 调性 + 语言 + 长度」指引。

## 背景与现状诊断

当前 `submit.js:125` 构造 `product = { ...config.product, utm_url }`，所有站点共用一份 `config.product`，唯一按站点变的只有 UTM 追踪参数。填充时 description 永远是 `long_description || description` 一份固定文本。

一份内容投所有站，压低通过率的三个机制：
1. **相关性错配**——AI 站要看 AI 能力、SaaS 目录要看商业价值、开发者站要看技术栈、indie 社区要看独立开发故事；同一段话对各站「想要的东西」命中率不同，审核判「不够贴合」。
2. **模板痕迹 = spam 信号**——一字不差投几十站，审核与反垃圾系统识别为批量模板，降低收录质量。差异化本身就是 anti-spam 信号。
3. **话术调性错位**——正式目录吃结构化表达，社区吃对话口吻；语气不对，收录后点击转化差。

两个现成但未利用的钩子：
- recipe 已有「按站点映射字段值」机制（`category` / `pricing` map），证明这套能跑通，只是没用到自由文本话术。
- `lang` 字段 258 站全标了（en 197 / zh 57 / multi 4），但内容不分语言——中文站现在大概率也填英文文案。

## 字段设计（已与用户确认）

targets.yaml 每站**只加一个 `niche` 字段**。一个 niche 同时承载「卖点角度 + 调性 + 推荐长度」（后两者作为 niche 的隐含属性写进生成指引，不单独存字段）。复用已有 `lang` 决定语言。

`niche` 枚举（6 个 + `design` 待定）：

| niche | 站点例子 | agent 生成时强调 | 调性 | 推荐长度 |
|---|---|---|---|---|
| `ai-tools` | Future Tools、AI 导航站 | 这是个 AI 工具，用什么 AI 能力解决什么具体任务 | 卖点直接 | tagline + 一段 |
| `saas` | SaaS 产品目录 | 产品定位、目标客户、业务问题、定价模式 | 正式、结构化 | 一段 |
| `devtools` | 开发者工具站、awesome-list | 技术栈、面向开发者、API/集成/工作流 | 技术向、具体 | awesome 要一行；其余一段 |
| `startup` | BetaList、startup88 类 | 新产品、独立开发者做的、解决什么痛点、新颖性 | 叙事感、indie 故事 | 一段 |
| `community` | Reddit、indie 论坛 | 真实分享口吻、给价值、不硬广 | 口语、克制推销 | 短、对话感 |
| `general` | 综合目录（兜底） | 清晰的一句话定位 + 核心功能 | 中性、信息密度高 | 一段 |
| `design`（待定） | 设计/灵感站 | 视觉、创意、灵感角度 | 视觉向 | 一段 |

**不单设的字段及理由**（防过度设计）：
- `tone`：与 niche 高度共线（community 必口语、saas 必正式），单设冗余且让初级模型多判一个主观维度。折叠进 niche。
- `audience`：niche 已隐含，纯冗余。
- `desc_length`：同 niche 隐含属性，且 agent 提交时能现场读表单 maxlength，比预存更准。不存死数据。

## 分类成本（为什么初级模型够用）

- AI 分组（`overseas_ai_directories` 86 + `chinese_ai_directories` 33 = 119）→ 规则映射 `ai-tools`，0 模型成本。
- `awesome_lists`(4) → `devtools`；`reddit`(10) + `communities_manual`(13) → `community`。规则映射，0 成本。
- `overseas_general`(59) + `overseas_directories`(39) + `chinese_general`(14) = **~112 站 `_unclassified`**，交 agent + haiku 逐站细分（saas/devtools/startup/design/general）。
- 未分类默认 `general`，agent 提交时 niche 缺失即当 general，保证不阻塞。

```javascript
// scripts/classify-niche.js — Rule-map a group key to a niche; CLI writes niche back to targets.yaml.
import { readFileSync, writeFileSync } from 'fs';
import { parseDocument } from 'yaml';

const ALLOWED = ['ai-tools', 'saas', 'devtools', 'startup', 'community', 'general', 'design'];

export function nicheForGroup(group) {
  if (/ai_director/i.test(group)) return 'ai-tools';
  if (group === 'awesome_lists') return 'devtools';
  if (group === 'reddit' || /communit/i.test(group)) return 'community';
  return '_unclassified';
}

function run({ apply }) {
  const path = 'targets.yaml';
  const doc = parseDocument(readFileSync(path, 'utf-8'));
  let added = 0, deferred = 0;

  for (const item of doc.contents.items) {       // top-level map: group -> seq
    const group = String(item.key);
    const seq = item.value;
    if (!seq || !seq.items) continue;
    for (const node of seq.items) {              // each node is a site map
      if (node.get('niche') != null) continue;   // never overwrite an existing niche
      const niche = nicheForGroup(group);
      node.set('niche', niche);
      if (niche === '_unclassified') deferred++; else added++;
    }
  }

  console.log(`rule-mapped: ${added}, deferred to agent (_unclassified): ${deferred}`);
  if (apply) {
    writeFileSync(path, doc.toString());
    console.log(`✓ wrote ${path}`);
  } else {
    console.log('(dry-run — pass --apply to write)');
  }
}

if (import.meta.url === `file://${process.argv[1]}`) {
  run({ apply: process.argv.includes('--apply') });
}

export { ALLOWED };
```

## Niche-driven Content (per-site copy)

Each target in `targets.yaml` carries a `niche` (and `lang`). Instead of submitting

2. Generate a description from `config.product` info, following the niche guide below.

| saas | positioning, target customer, business problem, pricing | formal, structured | one paragraph |
| devtools | tech stack, developer-facing, API/integration/workflow | technical, specific | one line for awesome-lists, else a paragraph |
| startup | new product, indie-made, the pain it solves, novelty | narrative, indie story | one paragraph |
| community | genuine sharing voice, give value, no hard sell | casual, restrained | short, conversational |
| general | a crisp one-line positioning + core features | neutral, information-dense | one paragraph |
| design | the visual/creative/inspiration angle | visual-leaning | one paragraph |

Pure-CLI users (no agent) can skip this — `submit` without `--description-file`
falls back to `config.product`, unchanged behavior.

## Plan Formation and Decision Rationale

- **Problem statement:** 同一产品向所有目录站提交同一份文案，相关性错配 + 模板痕迹 + 话术错位三重压低通过率。
- **Options considered:**
  - A 预设变体库（手写多份，零成本可控，但加产品费力）
  - B 规则模板拼装（自动但机械感重）
  - C 代码内调 LLM API 生成（质量高但每站扣费 + 要管缓存/兜底）
  - **D agent 运行时生成（用户提出）**：代码退回供料+通道，生成外包给运行项目的 Claude Code/Codex
- **Final decision:** D + 最小数据字段（单 `niche` + 复用 `lang`）+ generic/recipe 路径靠 `submit_text` 统一接入 + fallback 到 config。
- **Why this option:** D 零额外 API 成本（复用已付费的 agent 订阅）、契合项目既有 agent-first 架构与跨 agent 通用化协议、运行时上下文比固定 prompt 更贴合；字段最小化避免过度设计、降低初级模型分类负担。
