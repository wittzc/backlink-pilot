# Backlink Pilot 提交失败自我修正闭环 Implementation Plan

**Goal:** 把 backlink-pilot 从"提交失败只记账不反哺"升级为"失败自动剔除候选池"的自我修正系统，让 `auto: yes` 池随每轮跑动自动收敛到真正可自动提交的站点子集。

**Architecture:** 失败回写闭环。adapter 抛错携带 `code` → submit dispatcher 接 `code` → 裁决器按 code 写回 `targets.yaml`；ambiguous code 用 streak 门槛保护瞬时故障；外加 `locked` / `unlock` 命令让被锁清单可被诊断和单条恢复。

**Tech Stack:** Node.js ESM、commander CLI、bb-browser、yaml 2.8 (parseDocument)、proper-lockfile。

**验证方式（必填，必须可观测）:**
- `node --check` 5 个修改文件全部通过
- inline 测试 `getFailureStreak` + `applyFailureVerdict` 五个场景（streak 计数 / no-entry / already-applied / no-rule / dead 粘性）
- 真跑 5 站 smoke test：错误 code 准确、verdict 字段写入正确、streak 门槛生效
- 真跑 10 站第二轮：候选池有可观测剔除（`auto: yes` 计数下降）
- `node src/cli.js locked` 显示按 code 分桶，`unlock` 二次确认

## Plan Formation and Decision Rationale

- **Problem statement:**
  - 今日提交 23 个站点，成功 4 个失败 19 个，成功率 17%
  - 失败集中在 iframe 表单（Typeform / Tally 嵌入）和 SPA 动态表单
  - 更糟的是失败信号不反哺 `targets.yaml`——`auto: yes` 池里永远混着这些跑不通的站，下次批量还会被选进来再失败一次

- **Options considered:**
  - A. 写专用 iframe / SPA adapter — 治本但工程量大，且部分站 ROI 低；先做不解决"垃圾候选混在 auto:yes 池"问题
  - B. 手工维护 targets.yaml — 不可持续，每轮跑完都要人工筛
  - C. 失败回写闭环——adapter throws 携 code → 裁决表 → 写回 targets.yaml

- **Final decision:** C

- **Why this option:**
  - A 是后续工作，先有 C 的候选池治理才能让 A 的工作排期变得可见、可量化（`locked --code IFRAME_FORM` 直接产出 adapter 待办清单）
  - 工程量小（~200 行代码改动）、立刻可见效果
  - 与 A/B 不冲突，是 A 的前置基础设施

---

## Implementation Steps

1. **generic.js**：6 处 throw 升级为 `Object.assign(new Error(...), { code })`；新增第 5 轮 `long-wait`（8s）保守扫描兜底 SPA 慢渲染
2. **submit.js**：`classifyError` 改为优先读 `e.code`；新增 `runVerdict()` 在 3 个失败点（预检 404 / 预检 5xx / catch）末尾调用裁决器
3. **targets.js**：`VERDICT_TABLE`（7 个 code 的处置规则）+ `applyFailureVerdict(site, code)`（幂等、字段级防退化）
4. **tracker.js**：`getFailureStreak(site, code)` 反向遍历 submissions.yaml 统计末尾连续同 code 失败次数
5. **cli.js**：`--no-auto-verdict` 旁路开关
6. **bug fix**：脏数据 `status:dead + auto:yes` 无法补救——硬短路改为字段级防退化
7. **配套工具**：`locked [--code]` 列出被锁清单、`unlock <site>` 单条解锁

## 决策推导

**评估过的方案**：
- 写专用 iframe / SPA adapter（治本但工程量大，且无法解决"垃圾候选混在 auto:yes 池"问题）— 否定（应作后续工作，依赖本方案的待办清单）
- 手工维护 targets.yaml — 否定（不可持续，违背"重复 3 遍就 AI 化"原则）

**采用方案**：失败回写闭环 + streak 门槛 + 字段级防退化裁决器

**未来参考**：
- 看到「批量任务大量失败」时，**先问失败信号有没有反哺到任务源**——治理永远比扩大覆盖优先
- streak 门槛是处理「瞬时故障 vs 结构故障」混淆的标准手法（CHROME_TIMEOUT / SERVER_ERROR 等不入裁决表，瞬时不锁）
- 字段级防退化（safeOp 删退化字段而非整体短路）是 idempotent 状态机写入的更鲁棒模式
- 候选池治理可量化「下一步攻坚方向」：跑两轮看 `locked --code` 分桶分布，决定 iframe adapter 的 ROI 排序
