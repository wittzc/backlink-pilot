## Executive Assessment

当前项目可以支撑「批量外链提交」这件事，但还不能安全、稳定地直接跑 180 个站点。

已经修掉的 `bb-browser --tab`、snapshot 解析、动态等待、滚动扫描，是必要修复。没有这些地基，任何批量提交都会出现串 tab、空页面、误判字段的问题。这个方向是对的。

但如果继续沿着「增强 generic adapter，让它覆盖大多数站点」走下去，收益会快速下降。原因不是等待时间不够，而是网页表单差异本身已经超出 generic 的职责：

- React/Vue 页面可以通过等待和滚动救回一部分，但救不回额外业务字段，例如 category、pricing、tags、submitter role、logo upload。
- Typeform、Paperform、Tally、Airtable 这类 iframe/provider 表单不是普通 DOM 表单，generic 不应该硬闯。
- CAPTCHA、登录、付费、关闭提交入口属于策略问题，不是选择器问题。
- 不同站点对描述长度、产品分类、定价、提交人信息的要求不同，需要站点级数据映射。

所以更好的路线不是二选一地押注 generic 或 180 个手写 adapter，而是在两者中间加一层 `Recipe Adapter`。它把「稳定但字段稍复杂的表单」从 JavaScript adapter 中抽出来，用 YAML/JS 配置声明字段、选择器、值映射、提交按钮和验证规则。

## Plan Formation and Decision Rationale

- Problem statement:
  - 用户的真实目标是批量提交产品外链，不是单站自动化演示。
  - `targets.yaml` 中约 180 个 `auto: yes` 站点质量参差不齐，包含传统 HTML 表单、React 动态表单、iframe provider、登录墙、付费提交、失效页面。
  - 仅靠 generic adapter 无法稳定处理多数站点；为每个站点写完整 JS adapter 又会制造维护负担。
- Options considered:
  - Option A: 继续增强 generic adapter，加入更多正则、等待、滚动、iframe 探测。
  - Option B: 为所有高失败率站点逐个写 site-specific adapter。
  - Option C: 建立 `triage -> generic/recipe/provider/site-specific -> batch executor` 的分层架构。
- Final decision:
  - 采用 Option C。
- Why this option:
  - generic 继续保留，用来处理真正简单的可见表单。
  - recipe 覆盖「字段稳定、只是需要 selector 和 value mapping」的中等复杂表单。
  - provider adapter 覆盖 Paperform/Tally/Typeform/Airtable 等可复用流程。
  - site-specific adapter 只用于 Futurepedia、Product Hunt 等高价值且流程特殊的网站。
  - batch executor 负责节奏、去重、记录、失败分类，避免把运行策略散落到 adapter 中。

## Architecture Decision

目标架构分为六层。

1. **Target Registry Layer**
   - 输入仍然是 `targets.yaml`。
   - 每个 target 应逐步补充 `priority`、`value_tier`、`submit_type`、`provider`、`recipe`、`manual_reason` 等元数据。
   - 目标是让批量执行可以基于 metadata 排队，而不是每次重新猜。

2. **Triage Layer**
   - `src/triage.js` 继续作为准入层。
   - HTTP triage 用于全量粗筛。
   - browser triage 用于提交前精筛。
   - 新增分类建议：`recipe-ready`、`provider-ready`、`custom-adapter-needed`。

3. **Generic Adapter Layer**
   - 保留在 `src/sites/generic.js`。
   - 只处理可见、少字段、无额外业务逻辑的普通表单。
   - 不继续承担 iframe/provider、复杂分类映射、登录/CAPTCHA 处理。

4. **Recipe Adapter Layer**
   - 新增 `src/sites/form-recipe.js`。
   - 新增 `src/sites/recipes.js` 或 `recipes/*.yaml`。
   - 用声明式 recipe 描述字段选择器、字段值、select/radio mapping、按钮选择器、提交前验证和成功判定。
   - 适用对象：Future Tools、AI Valley、普通 WordPress Contact Form 7、稳定 React 表单。

5. **Provider Adapter Layer**
   - 新增或扩展 `src/sites/providers/*.js`。
   - Paperform、Tally、Typeform、Airtable 分开处理。
   - 优先策略是提取 iframe `src` 并直接打开 provider URL；如果 provider 需要一题一屏，就用状态机 adapter。
   - 不假设 bb-browser 的 CSS/fill 能跨 cross-origin iframe 直接操作。

6. **Batch Executor / Ops Layer**
   - `src/batch-submit.js` 或新 executor 负责任务队列。
   - 支持 dry-run、limit、category、priority、rate limit、retry limit、result store。
   - 每次执行必须记录成功、失败原因、截图或 DOM 证据、下一步动作。

## Why The Current Fix Route Is Only A Foundation

当前已修复的内容应该保留，它们解决的是浏览器自动化的基础可靠性：

- tab 绑定修复避免操作随机活跃标签页。
- snapshot 解析修复让 generic 能读懂真实 accessibility tree。
- polling 等待让 React/Vue 首屏空白不再被立即判失败。
- scroll 扫描能触发表单懒加载。
- provider 检测能把 iframe 表单从普通失败中分离出来。

但这些修复不应该继续无限加码成一个“万能 generic”。万能 generic 会出现三个问题：

- **可解释性差**：失败时不知道是字段找不到、业务字段缺失、iframe 限制、反自动化还是站点关闭。
- **误提交风险高**：正则猜错字段时，可能把产品名填到个人姓名，把 URL 填到错误字段。
- **维护成本高**：每加一个站点特例，都会影响其他站点的匹配行为。

因此当前路线需要从「修 generic」转成「分层自动化」。generic 只做简单站，recipe/provider/site-specific 分别吃掉不同复杂度的站点。

## Recipe Adapter Design

Recipe 的目标是让 Future Tools、AI Valley 这类站点不必写完整 JS adapter。

**Recipe 存储格式：直接 YAML（`recipes/<site-key>.yaml`），不做"先 JS 后迁 YAML"的两步走。**

理由：
- 用户角色是运营/产品出身的非开发，需要能直接编辑 recipe 文件添加新站点；JS object 在这个场景下是不必要的语言门槛。
- 实操经验：一旦 5+ 个 recipe 在 JS 里跑稳，迁 YAML 就会变成纯成本动作而被永久搁置。从一开始就 YAML 切断这个 trap。
- recipe 数据没有真函数需求（只有 selector + value mapping + 静态 default），YAML 表达力够用。

Recipe loader 在启动时 `js-yaml.load()` 解析为下述内存形态。下面的 JS object 示例展示的是 loader 解析后的运行时形状（也是单测里的 fixture 形状）：

```js
export const recipes = {
  futuretools: {
    url: 'https://www.futuretools.io/submit-a-tool',
    fields: [
      { key: 'submitter_name', selector: 'input[name="submitter_name"]', value: 'submitterName' },
      { key: 'name', selector: 'input[name="tool_name"]', value: 'name' },
      { key: 'url', selector: 'input[name="tool_url"]', value: 'utmUrl' },
      { key: 'description', selector: 'textarea[name="description"]', value: 'description' },
      { key: 'email', selector: 'input[name="submitter_email"]', value: 'email' }
    ],
    selects: [
      {
        key: 'category',
        selector: 'select[name="category"]',
        valueFrom: 'categories',
        map: {
          video: 'Generative Video',
          'developer-tools': 'Generative Code',
          default: 'Chat'
        }
      }
    ],
    radios: [
      {
        key: 'pricing_tier',
        name: 'pricing_tier',
        valueFrom: 'pricing',
        map: {
          free: 'free',
          freemium: 'freemium',
          'open-source': 'open_source',
          default: 'free'
        }
      }
    ],
    checkboxes: [
      // type 限定 {tos, privacy}；newsletter/marketing 不允许出现，loader 校验拒载
      { key: 'agree_tos', type: 'tos', selector: 'input[name="agree_tos"]' }
    ],
    submit: 'button[type="submit"]',
    success: {
      textIncludesAny: ['thank', 'submitted', 'received']
    }
  }
};
```

Recipe engine 需要提供的最小能力：

- `resolveRecipeValue(product, valueKey)`：从产品配置中取 name/url/utmUrl/email/description/categories/pricing。
- `fillTextFields(page, recipe.fields, product)`：稳定填 input/textarea。
- `fillSelects(page, recipe.selects, product)`：处理 select value/label。
- `fillRadios(page, recipe.radios, product)`：处理 radio by name/value。
- `fillLegalCheckboxes(page, recipe.checkboxes)`：仅勾 type ∈ {tos, privacy} 的 checkbox；其他 type 在 loader 阶段已被拒载，runtime 不再做白名单判断。
- `assertRequiredFields(recipe, product)`：提交前检测产品数据是否足够。
- `submitRecipe(page, recipe, { dryRun })`：dry-run 时只填表不提交。
- `readBackRecipeValues(page, recipe)`：用于 smoke test 读取回填结果。

## Provider Adapter Design

Provider adapter 不应作为 generic 的一部分。

建议优先顺序：

1. **Paperform**
   - 已在 AIToolsDirectory 样本中出现。
   - 通常 iframe `src` 可提取并直接打开。
   - 字段可能是多页但结构相对稳定，优先级高。

2. **Tally**
   - 很多目录站直接嵌入或跳转 Tally。
   - 可用 direct form URL，适合做 provider adapter。

3. **Typeform**
   - 常见一题一屏，需要按问题文本识别并点击下一步。
   - 实现成本高于 Paperform/Tally。
   - 只在命中多个高价值站点时投入。

4. **Airtable**
   - 可能存在反自动化和嵌入限制。
   - 优先标记 provider-ready，再选择样本验证。

bb-browser 是否能“切 iframe”不应作为主路线。跨域 iframe 对 CSS selector 和 DOM eval 都有天然边界，即便 CDP 可以附加 frame，也会让 generic 复杂度上升。更稳的做法是识别 provider、提取 `iframe.src`、直接打开表单 URL，或者写 provider 状态机。

## Batch Submission Policy

批量提交必须按运营队列执行，而不是一次性把 180 个站点全部丢给浏览器。

推荐规则：

- 每批 5-10 个站点。
- 每站最多自动重试 1 次；失败原因相同则停止。
- 每站提交前做 dry-run/preflight，确认字段可填。
- `CAPTCHA`、登录、付费、关闭入口直接进入 manual，不做绕过。
- provider/site-specific adapter 成功率稳定后再加入批量队列。
- 每次真实提交记录结果：target key、url、adapter type、status、error code、submittedAt、evidence。
- 对高价值站点保留 screenshot 或 DOM success text，方便人工复核。

## Non-Goals

- 不绕过 CAPTCHA（含 Cloudflare Turnstile / hCaptcha / reCAPTCHA / image challenge）；检测到一律 fail-fast 进 `manual-review`。
- 不自动处理需要登录或付费的提交入口。
- 不一次性为 180 个站点写 JavaScript adapter。
- 不把 cross-origin iframe 强行塞进 generic。
- 不在未 dry-run 验证字段映射前执行真实批量提交。
- **Checkbox 政策分级**：Newsletter / marketing opt-in checkbox **永不自动勾**；ToS / Privacy Policy checkbox 仅在 recipe 文件**显式声明**且 `type` ∈ `{tos, privacy}` 时勾，未声明的 checkbox 一律不勾。理由：完全禁勾会让多数 recipe 站点因必填 ToS 而 fail；完全代勾在批量场景属合规风险。分级策略要求 recipe 作者明确把 ToS/Privacy 列入 recipe，给出可追溯的同意路径。
- 不跨账号或跨产品复用 `submissions.yaml`；dedup 键只在当前 product config 范围内有效。
