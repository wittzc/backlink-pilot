# ADR: 分层适配器架构（triage → generic / recipe / provider / site-specific → batch）

- 状态: Accepted (2026-04-27)
- 来源: [plans/2026-04-27-批量外链提交架构评估与Recipe方案.md](../plans/2026-04-27-批量外链提交架构评估与Recipe方案.md)（合并升格自该 plan 的 inline ADR-001~005）
- 关联: [硬墙一律 fail-fast 不绕过](2026-04-27-硬墙一律fail-fast不绕过.md)

## Context

目标是批量提交约 180 个 `auto:yes` 站点，不是单站自动化演示。triage 数据显示失败不是单点 bug，而是站点生态的结构性差异：

- `triage-http.json`（total 180）：`manual-review=114`、`adapter-needed=44`、`iframe-provider=18`、`dead=4`
- 站点形态横跨：传统 HTML 表单、React/Vue 动态表单、Typeform/Tally/Paperform/Airtable 等 iframe provider、登录墙、付费提交、失效页面

核心判断：网页表单差异已超出 generic adapter 的职责边界。React 页面救得回首屏却救不回 category / pricing / tags 等业务字段；iframe provider 不是普通 DOM；CAPTCHA / 登录 / 付费是策略问题，不是选择器问题。

## Decision

采用六层架构，每层只吃一种复杂度：

1. **Target Registry** — `targets.yaml` 携带 `priority` / `value_tier` / `submit_type` / `provider` / `recipe` / `manual_reason` 元数据，让批量执行基于 metadata 排队而非每次重猜。
2. **Triage** — `src/triage.js` 作准入层，HTTP 全量粗筛 + browser 提交前精筛，分类含 `recipe-ready` / `provider-ready` / `custom-adapter-needed`。
3. **Generic Adapter** — `src/sites/generic.js`，只处理可见、少字段、无额外业务逻辑的普通表单，不再承担 iframe / 复杂映射 / 登录 / CAPTCHA。
4. **Recipe Adapter** — `src/sites/form-recipe.js` + `recipes/*.yaml`，声明式描述选择器、值映射、select/radio、提交按钮、成功判定，吃字段稳定的中等复杂表单。
5. **Provider Adapter** — `src/sites/providers/*.js`，Paperform / Tally / Typeform / Airtable 分开处理，优先提取 iframe `src` 直接打开 provider URL。
6. **Batch Executor / Ops** — `src/batch-submit.js` 负责队列、节奏、去重、retry、结果记录，把运行策略集中，不散落到 adapter。

## Alternatives Considered

- **Option A：继续增强 generic 成「万能适配器」**（加更多正则 / 等待 / 滚动 / iframe 探测）。否定，因为万能 generic 有三个不可逆问题：可解释性差（失败时分不清是字段找不到、业务字段缺失、iframe 限制还是站点关闭）、误提交风险高（正则猜错会把产品名填进姓名字段）、维护成本随站点数膨胀（每加一个特例影响其他站点匹配）。
- **Option B：为所有高失败站逐个写 site-specific JS adapter**。否定，180 站逐个手写的维护负担不可控。

## Consequences

- 正面：失败可归因到具体层；加站点的默认路径是写一条 YAML recipe 而非 JS adapter（非开发用户也能加）；运行策略集中在 executor，可独立演进节奏 / 去重 / retry。
- 成本：引入多层路由复杂度；triage 成为批量执行的前置依赖；site-specific adapter 仅保留给高价值且流程特殊的站点（如 Product Hunt 类）。
