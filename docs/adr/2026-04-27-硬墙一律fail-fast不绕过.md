# ADR: 硬墙（CAPTCHA / 登录 / 付费）一律 fail-fast，不绕过

- 状态: Accepted (2026-04-27)
- 来源: [plans/2026-04-27-批量外链提交架构评估与Recipe方案.md](../plans/2026-04-27-批量外链提交架构评估与Recipe方案.md)（升格自该 plan 的 inline ADR-007 + Non-Goals + Batch Policy）
- 关联: [分层适配器架构](2026-04-27-分层适配器架构.md)

## Context

triage 中占比最大的 `manual-review` bucket（180 站里 114 个）主要是「硬墙」站点：Cloudflare Turnstile / hCaptcha / reCAPTCHA / image challenge、登录墙、付费提交、已关闭提交入口。

这些是策略问题，不是选择器问题。再强的 adapter 也无法把它们变成可自动填的表单。如果让 adapter 去「闯」，结果是失败信号被污染：分不清是字段没找到还是被硬墙挡住。

## Decision

检测到硬墙一律 **fail-fast** 进 `manual-review`，带 reason 子分类，不做任何绕过尝试：

- `manual-review.reason` 枚举：`captcha-required` / `login-required` / `paid` / `closed-submission` / `unknown`
- CAPTCHA 检测信号至少包含：`iframe[src*="challenges.cloudflare.com"]`、`.h-captcha`、`.g-recaptcha`、`script[src*="recaptcha"]`、`script[src*="hcaptcha"]`
- **优先级**：CAPTCHA 检测优先于 recipe-ready。硬墙不能被 adapter 覆盖，即使某站有可用 recipe，检测到硬墙也归 manual-review。

## Alternatives Considered

- **接入 2Captcha / Anti-captcha 等付费打码服务自动过验证码**。否定，两个理由：（1）与本项目「免费工具」定位直接冲突；（2）引入按量扣费的第三方 API，是项目刻意规避的成本与复杂度来源。

## Consequences

- 正面：成功率信号干净（失败归因明确）；不烧钱；不触碰各站反自动化条款的合规灰区。
- 成本：硬墙站点（占当前池子大头）永远进 manual，需人工或 Computer Use 处理，构成自动化覆盖率的天花板。这是有意接受的边界，不是缺陷。
- 实操呼应：真实提交日志（[research/2026-05-04-外链提交过程日志.md](../research/2026-05-04-外链提交过程日志.md)）印证了这条边界——高命中路径是无验证码的第三方真表单 / 可直接 POST 的 HTML 表单 / 公开 JSON API；reCAPTCHA / 登录 / 付费 / 强制 badge / 强制上传一律跳过。
